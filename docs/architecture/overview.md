### 请求流向

**用户端请求流向：**
```
用户操作 → Expo App → BFF (/api/v1/*) → 算法后端
                  ↓
               数据库
                  ↑
用户展示 ← Expo App ← BFF ← 算法后端
```

**管理后台请求流向：**
```
管理员操作 → Admin Panel → BFF (/api/admin/*) → 数据库
                                    ↓
                              审计日志记录
```

**关键点：**
- 前端不直接与算法后端通信，所有请求都经过 BFF 中转
- 管理后台与用户端共享同一个 BFF 服务，通过路由组隔离
- 管理后台所有操作自动记录审计日志

| WebSocket | gorilla/websocket | 实时通信 |

### BFF 层职责

| 职责 | 说明 |
|------|------|
| 用户认证 | 注册、登录、JWT Token 管理 |
| 用户信息 | 头像、昵称等基本信息 |
| 用户偏好 | 主题、通知设置等个性化配置 |
| 账户余额 | 余额查询、扣费、充值、交易记录 |
| 消息读取 | 只读访问 chat_schema（由算法后端写入） |
| 算法服务代理 | 转发前端请求到算法后端，处理响应结果 |
| WebSocket 网关 | 维护与前端的实时连接，中转算法后端消息 |
| **管理后台** | 仪表盘、用户/订单/公告管理、系统配置、审计日志 |

### 认证与鉴权

```
用户端请求:
  Request → AuthMiddleware (JWT校验) → Handler

管理后台请求:
  Request → AuthMiddleware (JWT校验) → AdminMiddleware (角色校验) → Handler
                                            │
                                            ├── role = "admin" → 允许
                                            ├── role = "super_admin" → 允许
                                            └── role = "user" → 403 拒绝
```

- 管理员复用现有 User 模型，新增 `role` 字段区分角色
- JWT Claims 中携带 `role`，AdminMiddleware 据此判断权限
- 所有管理操作自动写入 `admin_audit_logs` 表

## 数据层

详见 [数据库设计文档](./database.md)

### PostgreSQL

与算法后端共享同一个 PostgreSQL 实例，通过 Schema 隔离：

| Schema | 权限 | 说明 |
|--------|------|------|
| `bff_schema` | BFF 读写 | 用户表、余额表、配置表等 |
| `chat_schema` | BFF 只读 | 消息表、会话表（算法后端写入） |

**表结构概览：**

```
bff_schema                   chat_schema
├── users (+role)            ├── conversations
├── user_preferences         └── messages
├── referrals
├── system_messages
├── token_usage
├── user_balance             ← 账户余额功能
├── transactions             ← 账户余额功能
├── recharge_orders          ← 账户余额功能
├── admin_audit_logs         ← 管理后台功能
└── system_configs           ← 管理后台功能
```

### Redis

Redis 在 100W 并发场景下承担关键角色：

| 职责 | 说明 |
|------|------|
| WebSocket 连接状态 | 记录用户连接到哪个 BFF 实例 |
| 用户在线状态 | 实时追踪在线用户 |
| Pub/Sub 消息中转 | 多 BFF 实例间的消息同步 |
| Token 管理 | JWT 黑名单、Refresh Token 存储 |
| 热点数据缓存 | 用户信息、偏好设置等高频读取数据 |
| API 限流 | 防止恶意请求，保护系统稳定性 |
| 余额查询缓存 | balance:{user_id} 短期缓存（60s） |
| 系统配置缓存 | config:* 配置项缓存，更新时失效 |

**为什么需要 Redis：**
- 100W 并发需要多个 BFF 实例水平扩展
- 用户 A 连接 BFF-1，用户 B 连接 BFF-2 时，需要 Redis Pub/Sub 中转消息
- 减少 PostgreSQL 压力，热点数据走缓存
- 系统配置变更后通过缓存失效机制即时生效
