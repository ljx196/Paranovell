# 管理后台 PRD

## 1. 功能概述

### 1.1 背景

GenNovel 平台已完成用户端核心功能（认证、聊天、余额管理、系统消息），但缺乏运营管理能力。目前所有管理操作（用户封禁、发送公告、调整余额等）只能通过直接操作数据库完成，效率低、风险高、无法审计。需要一套完整的管理后台来支撑日常运营。

### 1.2 目标

- 提供平台运营数据的实时总览，辅助运营决策
- 实现用户全生命周期管理（查看、封禁、调账、重置密码）
- 提供充值订单和交易流水的查询与管理能力
- 支持管理员在线发送系统公告，无需操作数据库
- 平台关键配置可在线调整，无需修改代码和重启服务
- 所有管理操作留痕，支持安全审计

### 1.3 架构方案

采用**方案 A：独立管理前端 + 共享 BFF**。

```
┌─────────────────────┐     ┌─────────────────────┐
│    用户端前端         │     │    管理端前端         │
│  Expo/RN (Web/App)  │     │  React + Ant Design  │
│  端口 :8081         │     │  端口 :3001          │
└─────────┬───────────┘     └─────────┬───────────┘
          │                           │
          │     ┌─────────────────┐   │
          └────►│    BFF (Go/Gin) │◄──┘
                │    端口 :8080   │
                │                 │
                │  /api/v1/*      │ ← 用户端 API（AuthMiddleware）
                │  /api/admin/*   │ ← 管理端 API（AuthMiddleware + AdminMiddleware）
                └────────┬────────┘
                         │
                ┌────────┴────────┐
                │   PostgreSQL    │
                │   Redis         │
                └─────────────────┘
```

**关键设计决策：**

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 前端框架 | React + Ant Design Pro | 管理后台生态最成熟，组件丰富，表格/表单开箱即用 |
| 是否独立部署 | 是，独立端口 | 与用户端完全隔离，互不影响 |
| 是否共享 BFF | 是，同一个 Go 服务 | 复用 Service 层和数据模型，减少重复代码 |
| 权限区分方式 | User 表加 `role` 字段 + JWT 含角色声明 + AdminMiddleware | 简单直接，满足当前需求 |

### 1.4 管理员角色定义

| 角色 | role 值 | 权限范围 |
|------|---------|----------|
| 普通用户 | `user` | 仅用户端功能 |
| 管理员 | `admin` | 全部管理后台功能 |
| 超级管理员 | `super_admin` | 管理后台 + 管理员账号管理 |

> 初期仅实现 `user` 和 `admin` 两个角色，`super_admin` 留作后续扩展。首个管理员账号通过数据库手动设置 `role='admin'`。

## 4. P0 功能详细设计

### 4.1 管理员认证与鉴权

#### 4.1.1 登录

管理员使用与用户端相同的账号密码登录，BFF 在 JWT 中附带 `role` 字段。管理端前端在登录后检查 `role`，非 admin 角色则拒绝进入。

**流程：**

```
管理端前端                      BFF
    │                           │
    │  POST /api/v1/auth/login  │
    │  {email, password}        │
    │ ─────────────────────────>│
    │                           │  ← 复用现有登录接口
    │  {token, user: {role}}    │
    │ <─────────────────────────│
    │                           │
    │  检查 role == 'admin'     │
    │  否 → 显示"无管理权限"     │
    │  是 → 进入管理后台         │
```

> 复用现有 `/api/v1/auth/login` 接口，无需新增登录接口。只需在返回的用户信息中包含 `role` 字段。

#### 4.1.2 AdminMiddleware

所有 `/api/admin/*` 路由前置 AdminMiddleware：

```
1. 执行 AuthMiddleware（验证 JWT，提取 userID）
2. 执行 AdminMiddleware：
   - 从 JWT Claims 中读取 role
   - role != 'admin' && role != 'super_admin' → 403 Forbidden
   - 通过 → 放行，将 adminID 写入 Context
```

#### 4.2.2 指标卡片定义

| 指标 | 计算方式 | 数据来源 |
|------|----------|----------|
| 总用户数 | `COUNT(*) FROM users WHERE deleted_at IS NULL` | users |

#### 4.3.3 用户操作

**封禁/解封用户**

| 项目 | 说明 |
|------|------|
| 触发 | 点击「封禁」/「解封」按钮 |
| 确认 | 弹窗二次确认，需填写原因 |
| 效果 | 修改 users.status（1→0 封禁，0→1 解封） |
| 后续 | 被封禁用户下次请求时 AuthMiddleware 返回 403 |
| 审计 | 写入 admin_audit_logs |

**手动调账**

| 项目 | 说明 |
|------|------|
| 触发 | 点击「调账」按钮 |
| 弹窗 | 类型（增加/扣除）、金额、原因（必填） |
| 效果 | 更新 user_balance，写入 transactions（type=gift 或 refund） |
| 校验 | 扣除时不能超过当前余额 |
| 审计 | 写入 admin_audit_logs，detail 中记录调账前后余额 |

**重置密码**

| 项目 | 说明 |
|------|------|
| 触发 | 点击「重置密码」按钮 |
| 确认 | 弹窗二次确认 |
| 效果 | 生成随机密码，更新 users.password_hash，发送密码重置邮件给用户 |
| 审计 | 写入 admin_audit_logs |

## 9. 验收标准（P0）

### 9.1 功能验收

| 编号 | 验收项 | 标准 |
|------|--------|------|
| A-01 | 管理员登录 | admin 角色可登录管理后台，普通用户被拒绝 |
| A-02 | 仪表盘数据 | 总用户数、今日活跃、今日新增、今日收入数据正确 |
| A-03 | 趋势图 | 7/30/90天趋势图数据正确、切换正常 |
| A-04 | 用户搜索 | 按邮箱/昵称关键词搜索结果正确 |
| A-05 | 用户筛选 | 按状态、注册时间筛选结果正确 |
| A-06 | 封禁/解封 | 封禁后用户无法访问 API，解封后恢复 |
| A-07 | 手动调账 | 增加/扣除后 balance、transactions 记录正确 |
| A-08 | 重置密码 | 密码重置后用户可用新密码登录，旧密码失效 |
| A-09 | 订单查询 | 按状态/支付方式/日期/用户筛选正确 |
| A-10 | 交易查询 | 按类型/日期/用户筛选正确 |
| A-11 | 发送公告 | 全部用户/指定用户均能收到消息 |
| A-12 | 配置修改 | 修改后配置立即生效，用户端行为随之改变 |
| A-13 | 操作日志 | 所有管理操作均有审计记录，detail 完整 |
| A-14 | 二次确认 | 封禁、调账、群发公告等操作有确认弹窗 |

### 9.2 非功能验收

| 编号 | 验收项 | 标准 |
|------|--------|------|
| N-01 | 权限隔离 | 普通用户无法访问任何 /api/admin/* 接口 |
| N-02 | 响应时间 | 列表查询 < 500ms，仪表盘加载 < 1s |
| N-03 | 分页 | 大数据量下分页查询不卡顿 |
| N-04 | 审计完整性 | 不存在无日志记录的管理操作 |
